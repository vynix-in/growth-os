# Vynix 60-day growth plan

This is the plan to get Vynix in front of as many of the right people as possible
over the next 60 days. It is built around what this system produces and what a
person needs to do alongside it. The goal is simple: more developers find Vynix,
try it, and tell others.

## Where growth comes from

There are three engines, and we are running all three.

1. Search. People look for "bug reporting tools", "Marker.io alternative", "how
   to give an AI agent context", and similar. We have pages that answer those
   searches: the blog, the head-to-head comparisons, the buyer guides, and the
   use-case pages. Search is slow to start but compounds.
2. Distribution. We do not wait for search. Every blog post becomes a set of
   ready-to-paste drafts for X, LinkedIn, Reddit, dev.to and Hashnode. Posting
   where developers already gather brings traffic now and earns backlinks that
   help search later.
3. Presence. Directory listings, GitHub repositories, and outreach to
   newsletters and communities put Vynix where people browse for tools. Each one
   is a link back and a new way to be found.

## What is already live

- A resources site with the blog, comparisons, buyer guides, use-case pages and
  a help center, all built for search with proper structured data.
- Eleven public repositories in the GitHub organisation, each linking back to
  Vynix.
- A reviewer that checks every page before it ships, and a publication gate that
  blocks anything sensitive.
- A dashboard that shows what is published and tracks the 60-day targets.

The dashboard shows progress against each of these.

## The weekly rhythm

Week 1 and 2. Get the base live and start posting. Publish the site, submit the
first ten directories, and post one piece of content per working day using the
distribution drafts. Reply to every comment quickly.

Week 3 and 4. Widen the search coverage. Add more comparison and buyer-guide
pages for the searches people actually use. Begin outreach to two or three
newsletters and one podcast. Keep posting daily.

Week 5 and 6. Lean into what works. Look at which posts and which platforms
brought traffic, and make more of that. Pitch a guest post to one high-authority
site. Ask happy users for a quiet word of mouth.

Week 7 and 8. Compound. Refresh the strongest pages, add internal links between
related content, and push the open-source repositories for stars so they can
reach the GitHub trending lists. Submit the remaining directories.

## What the founder does

Most of this runs on its own. The parts that need a person are the ones that
should need a person: posting to third-party sites, submitting to directories,
and reaching out to people. The dashboard lists exactly what is waiting, with the
drafts ready to copy and paste. Spend thirty minutes a day on it and the system
handles the rest.

## How we will know it is working

Watch four things: pages getting indexed, branded searches for "Vynix" rising,
referral traffic from the communities we post in, and stars on the open-source
repositories. Early on, distribution moves first. Search follows over the weeks.

## The one decision that unlocks more

The resources site is currently hosted on a separate property so it could go live
safely without touching the production site. Moving it onto the main domain, on a
subdomain or a path that does not clash with the existing app, would send all of
this search value straight to the main domain. That is the highest-leverage next
step, and it is a hosting decision for the founder to make.
